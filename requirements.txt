3) How WIP Voucher works in your model (completion-only)
WIP Voucher = “Department Completion Voucher”
•	Worker enters:
•	SKU
•	Department
•	Completed Qty (pairs)
•	Labour
System does (backend):
•	Consumes only that department’s BOM RM for that qty
Cr Inventory (RM/SFG) at WAC
Dr Production Clearing
•	Books only that department’s labour
Cr Wages Payable
Dr Production Clearing
•	Adds to WIP dept pool
o	wip_dept_balance.qty_pairs += completed_qty
o	wip_dept_balance.cost_value += dept_cost_for_that_qty
ledger “IN” row created
Workers never see cost. Cost stays in DB.
4) How Finished Production works later (the “skip and subtract” you want)
When FG is posted for FG_qty (pairs), system loops each BOM department. For each dept:
•	available = wip_dept_balance.qty_pairs
•	use = min(FG_qty, available)
•	If use > 0 (already completed earlier)
✅ Don’t deduct RM/labour again.
Just reduce the pool:
•	wip_dept_balance.qty_pairs -= use
•	wip_dept_balance.cost_value -= allocated_cost(use) (FIFO or weighted avg behind the scenes)
•	ledger “OUT” row created
          If FG_qty > available (shortfall), for the remaining quantity (FG_qty - available):
•	do normal BOM consumption + labour posting for that dept (because it wasn’t completed earlier)
Finally
•	Capitalize FG:
•	Dr Inventory – FG
•	Cr Production Clearing
(total cost = costs newly incurred + costs pulled from pools)
That’s exactly your rule:
o	If FG < stitched qty → stitched balance remains
o	If FG > stitched qty → you’ll consume all stitched balance and do extra stitching cost for remaining

o	 Hidden backend pool (not shown to workers) to support delayed finishing without any Batch ID:
•	Maintain a table like dvc_pending_pool per (branch_id, sku_id, department_id) with:
o	pending_qty_pairs
o	pending_cost_amount
o	last_updated_at
•	This pool increases when you post the DVC/WIP completion voucher for that department, and decreases when Finished Production consumes it later.
o	Validation rules:
•	For DVC_ABANDON lines: abandon_qty_pairs <= pending_qty_pairs in dvc_pending_pool for that exact (branch, sku, department).
•	For RM/SFG/FG loss: loss_qty <= current_stock_qty (unless you allow negative stock, but you shouldn’t).
•	No cost fields are visible on screen for any worker role; costs are calculated + posted in backend only.

•	Always use cost (WAC or your chosen method) for GL + stock value. Never selling rate.
•	For DVC_ABANDON, calculate a unit cost from the pool:
o	unit_cost = pending_cost_amount / pending_qty_pairs (guard against divide-by-zero)
o	cost_to_clear = unit_cost * abandon_qty_pairs

o	Only the pending pool is reduced.

Case: Customer pays  100,000 online (not cleared yet)
Step 1 — Pending
•	Dr Bank Clearing 100,000
•	Cr AR 100,000 (or Cr Sales Revenue if it’s a direct receipt)
At this point:
•	Bank (real) didn’t increase yet ✅
•	Bank Clearing increased ✅
Total assets increased by 100,000 once (which is correct, because you now have a cheque/claim to cash).
Step 2 — When bank clears
•	Dr Bank 100,000
•	Cr Bank Clearing 100,000
Now:
•	Bank goes up ✅
•	Clearing goes down ✅
Net effect on assets = 0 at clearance time (you’re just moving value from “pending” to “actual bank”).
So nothing gets counted twice; it’s a transfer between two asset accounts.
UNIVERSAL RULES
A) One universal “Voucher Engine” (non-negotiable)
All vouchers (Sales, Purchase, Cash/Bank/JV, Stock Count, STN, Production, DCV, Abnormal Loss, RGP) must follow the same pattern:
•	voucher_header: voucher_no, voucher_type, branch_id, voucher_date, status (Draft/Pending/Approved/Rejected/Posted), created_by/on, approved_by/on, remarks
•	voucher_lines: item/account/labour reference + qty/rate/amount + department + extra JSON/meta
•	posting tables (system-generated):
o	gl_entry_header + gl_entry_lines
o	stock_ledger
Not all vouchers need status so keep status fields for every voucher, but allow “simple vouchers” to skip the workflow
•	status always exists, but voucher types can be configured as:
o	requires_approval = false
o	default_status_on_save = POSTED
•	Then for simple vouchers (like many cash vouchers), you save → it becomes POSTED immediately.
•	approved_by, approved_on can be NULL unless requires_approval = true.
